httpx ≥0.28
Cliente HTTP moderno para Python
Documentação oficial Quickstart Client avançado httpx.head() httpx.post()

Por que httpx e não requests? O requests é a biblioteca HTTP mais popular em Python, mas o httpx foi escolhido por três razões:

  • API idêntica ao requests — zero curva de aprendizado para quem já conhece requests
  • Suporte a HTTP/2 nativo — melhor desempenho em APIs modernas
  • Suporte assíncrono built-in — se o projeto crescer para async (FastAPI, etc.), não precisamos trocar a biblioteca
  • Timeout granular — permite controlar connect_timeout, read_timeout e write_timeout separadamente

Como usamos: httpx.post() — Overpass API

A Overpass API (OpenStreetMap) aceita a query OverpassQL via POST. Usamos o método síncrono simples:

src/fabrica_sites/agents/scout/sources/overpass.py 
def _post(self, query: str) -> list[dict]:
    """Tenta cada espelho da Overpass até um responder."""
    ultimo_erro: Exception | None = None
    for url in self.endpoints:
        try:
            resp = httpx.post(
                url,
                data={"data": query},      # form-data, não JSON
                timeout=self.timeout,       # 90s — a Overpass pode demorar
                headers={"User-Agent": config.USER_AGENT},
            )
            resp.raise_for_status()        # lança HTTPStatusError se 4xx/5xx
            return resp.json().get("elements", [])
        except Exception as exc:
            ultimo_erro = exc
            continue                       # tenta o próximo espelho
    raise RuntimeError(f"Todos os espelhos falharam. Último: {ultimo_erro}")
Por que data= e não json=?
A Overpass API foi projetada nos anos 2000 e aceita a query como form data (application/x-www-form-urlencoded), não JSON. O parâmetro data={"data": query} envia exatamente isso. json= mandaria Content-Type: application/json, que a Overpass não entende.

Como usamos: httpx.post() — Serper API

O Serper usa JSON moderno. A diferença está no json= e nos headers de autenticação:

src/fabrica_sites/agents/scout/sources/serper.py 
def _search(self, query: str) -> list[dict]:
    resp = httpx.post(
        _SERPER_URL,          # "https://google.serper.dev/maps"
        headers={
            "X-API-KEY": self._api_key,         # autenticação via header
            "Content-Type": "application/json",
        },
        json={                                   # serializa dict para JSON
            "q": query,
            "gl": "br",      # geolocalização: Brasil
            "hl": "pt-br",   # idioma da resposta
            "num": self._max_results,
        },
        timeout=15.0,
    )
    resp.raise_for_status()
    return resp.json().get("places", [])

Como usamos: httpx.head() — DomainGuesser

Para verificar se um domínio existe sem baixar o corpo da página, usamos HEAD. É ~10× mais rápido que GET:

src/fabrica_sites/agents/scout/enrichers/domain_guesser.py 
def _checar_dominio(dominio: str, timeout: float) -> bool:
    """HEAD request; True se responder com status < 400."""
    for scheme in ("https", "http"):     # tenta HTTPS primeiro
        try:
            resp = httpx.head(
                f"{scheme}://{dominio}",
                headers={"User-Agent": _UA},
                follow_redirects=True,   # segue 301/302 — importante!
                timeout=timeout,         # 3s por padrão
            )
            if resp.status_code < 400:
                return True
        except Exception:
            continue                     # qualquer erro = domínio não responde
    return False
Por que follow_redirects=True?
Muitos sites respondem ao domínio raiz com redirect 301 para www.dominio.com.br. Sem follow_redirects=True, o HEAD retornaria 301 que não passaria no teste < 400... e classificaríamos como "não existe". Com o follow ativo, seguimos até o destino final.
O que poderia melhorar
Criamos um novo cliente httpx por chamada de _checar_dominio. O ideal seria um Client compartilhado por thread, pois o Client mantém um pool de conexões e reutiliza conexões TCP. Para a escala atual (300 negócios) é aceitável. Em escala maior, mudaria para httpx.Client() como context manager dentro do ThreadPoolExecutor.

Pydantic v2 ≥2.0
Validação de dados e tipagem em runtime
Documentação v2 BaseModel Field() Serialização model_copy()

Pydantic transforma dicionários (dados brutos de API) em objetos Python com tipos validados. No Pydantic v2, o mecanismo interno foi reescrito em Rust — a validação é ~5-50× mais rápida que v1.

Nosso modelo central: Business

src/fabrica_sites/models.py 
class Business(BaseModel):
    """Negócio classificado e pontuado — a unidade central do Scout."""

    # Identidade (vinda do OSM)
    osm_type: str
    osm_id: int
    nome: str | None = None

    # Classificação de setor
    setor: str = "outros"
    setor_nome: str = "Outros"

    # Localização
    lat: float | None = None
    lon: float | None = None
    endereco: str | None = None

    # Contato
    telefone: str | None = None
    email: str | None = None
    website: str | None = None
    website_kind: WebsiteKind = WebsiteKind.NENHUM

    # Avaliação de oportunidade
    site_status: SiteStatus = SiteStatus.DESCONHECIDO
    score: int = 0
    score_label: str = "BAIXA"
    score_motivos: list[str] = Field(default_factory=list)
    contactavel: bool = False

    raw_tags: dict[str, str] = Field(default_factory=dict)

    @property
    def osm_url(self) -> str:
        return f"https://www.openstreetmap.org/{self.osm_type}/{self.osm_id}"
Por que Field(default_factory=list) e não = []?
Em Python, um argumento padrão mutável (como = []) é compartilhado entre TODAS as instâncias da classe — um bug clássico. Field(default_factory=list) instrui o Pydantic a chamar list() para cada nova instância, garantindo listas independentes. O Pydantic v2 já detectaria a lista muda e jogaria erro, mas é boa prática explicitar.

Enums como tipos de dado

Usamos str, Enum — os enums herdam de str para ser serializáveis sem conversão extra:

class SiteStatus(str, Enum):
    SEM_SITE = "SEM_SITE"
    SO_REDE_SOCIAL = "SO_REDE_SOCIAL"
    COM_SITE = "COM_SITE"
    DESCONHECIDO = "DESCONHECIDO"

Herdar de str significa que SiteStatus.SEM_SITE == "SEM_SITE" é True e o JSON gerado usa a string, não o objeto enum. Isso simplifica banco de dados, templates Jinja2 e comparações em testes.

model_copy() — imutabilidade nos enriquecedores

O DomainGuesser descobre um site e precisa atualizar o Business. Em vez de mutar o objeto existente, criamos uma cópia atualizada:

src/fabrica_sites/agents/scout/enrichers/domain_guesser.py 
def _rescore(b: Business, novo_website: str) -> Business:
    """Cria nova instância de Business com site descoberto e score recalculado."""
    campos = {
        "website_kind": WebsiteKind.PROPRIO,
        "setor": b.setor,
        "telefone": b.telefone,
        "email": b.email,
        "horario": b.horario,
        "endereco": b.endereco,
    }
    nova_aval = scorer_module.score(campos)
    return b.model_copy(update={        # cria nova instância, b original intacto
        "website": novo_website,
        "website_kind": WebsiteKind.PROPRIO,
        **nova_aval,                    # spread do dict de pontuação
    })
Por que imutabilidade aqui?
O enrich() recebe uma lista e a devolve modificada. Se mutarmos os objetos no lugar, o chamador original vê mudanças inesperadas (side effects). Com model_copy(), o Business original fica intacto e a função é previsível — característica de funções puras. Facilita testar, depurar e raciocinar.

monkeypatch — substituir rede por mock

O monkeypatch é uma fixture especial que substitui funções/atributos durante o teste e desfaz a substituição automaticamente ao final. Essencial para testes sem rede:

def test_checar_dominio_verdadeiro_para_status_200(monkeypatch):
    # Cria uma resposta falsa que sempre retorna 200
    class _FakeResp:
        status_code = 200

    # Substitui httpx.head pelo nosso mock (só durante este teste)
    monkeypatch.setattr(
        "fabrica_sites.agents.scout.enrichers.domain_guesser.httpx.head",
        lambda *a, **kw: _FakeResp(),
    )

    assert _checar_dominio("qualquerdominio.com.br", timeout=1.0) is True
    # Após o teste: httpx.head é restaurado ao original automaticamente
Por que monkeypatch em vez de unittest.mock?
O monkeypatch do pytest é mais simples para casos comuns. unittest.mock tem mais poder (verify call counts, assert_called_with, etc.) mas exige mais boilerplate. Para este projeto, monkeypatch é suficiente e mais legível.

concurrent.futures — ThreadPoolExecutor

Docs concurrent.futures ThreadPoolExecutor as_completed()

O DomainGuesser verifica ~300 domínios em paralelo. Sem threads, seria sequencial e levaria minutos:

from concurrent.futures import ThreadPoolExecutor, as_completed

with ThreadPoolExecutor(max_workers=15) as pool:
    # Submete todas as verificações ao pool
    futures = {
        pool.submit(_encontrar_site, b.nome, self.timeout): i
        for i, b in alvos.items()
    }
    # as_completed() itera na ordem de conclusão (não de submissão)
    for future in as_completed(futures):
        idx = futures[future]
        resultado = future.result()  # pode lançar exceção se a função falhou
        if resultado:
            encontrados[idx] = resultado
Thread vs Process
Usamos ThreadPool (não Process). Tarefas I/O-bound (esperar resposta HTTP) se beneficiam de threads porque a thread fica bloqueada aguardando rede — liberando o GIL para outras. Para CPU-bound (cálculos pesados), ProcessPoolExecutor seria melhor. DomainGuesser é puramente I/O, então threads são a escolha certa.

urllib.parse — urlparse para domínios

urlparse()

Usado no _is_social() do classifier para extrair o domínio real de uma URL (evitando o bug com substring):

from urllib.parse import urlparse

def _is_social(url: str) -> bool:
    try:
        netloc = urlparse(url).netloc.lower()
        # urlparse("https://www.salaox.com.br/page").netloc == "www.salaox.com.br"
        host = netloc.removeprefix("www.")
        # host == "salaox.com.br" — NÃO é x.com
        return any(host == dom or host.endswith("." + dom)
                   for dom in _SOCIAL_DOMAINS)
    except Exception:
        u = url.lower()
        return any(dom in u for dom in _SOCIAL_DOMAINS)  # fallback
O bug que isso corrigiu
O código original usava any(dom in url). O domínio "x.com" (Twitter/X) é substring de "salaox.com.br" — então um salão de beleza estava sendo classificado como rede social! O urlparse extrai o domínio real e resolve o problema na raiz.

hashlib — IDs sintéticos para o Serper

Docs hashlib 
import hashlib

def _serper_id(title: str, address: str) -> int:
    """ID sintético estável: MD5 de (título+endereço), truncado a 31 bits."""
    raw = f"{title}|{address}".encode("utf-8")
    h = hashlib.md5(raw).hexdigest()
    return int(h[:8], 16) & 0x7FFF_FFFF  # sempre positivo (bit de sinal zerado)

O OSM usa IDs inteiros reais. O Serper não tem IDs — então geramos um ID determinístico: o mesmo negócio sempre produzirá o mesmo ID, permitindo deduplicação entre execuções.

unicodedata + re — slugificação de texto

unicodedata re 
import unicodedata, re

def _ascii_slug(text: str) -> str:
    """Texto → slug ASCII: sem acentos, minúsculas, sem espaços."""
    # NFKD decompõe "é" em "e" + combining accent
    text = unicodedata.normalize("NFKD", text)
    # encode/decode joga fora os combining accents
    text = text.encode("ascii", "ignore").decode("ascii")
    # remove qualquer coisa que não seja letra, número ou espaço
    text = re.sub(r"[^a-zA-Z0-9\s]", "", text)
    # remove espaços
    return re.sub(r"\s+", "", text).lower()

# "Hotel Ilhas da Grécia" → "hotelilhasdagrecia"
# "Salão Beleza & Arte"   → "salaobelezaarte"

PyPA: src vs flat layout Hynek Schlawack — Testing & Packaging

Por que colocar o código em src/ e não na raiz? Este é um debate antigo na comunidade Python, mas a resposta moderna é clara:

  • Evita importação acidental: se o código ficasse na raiz (ex.: fabrica_sites/), um import fabrica_sites rodado DA raiz importaria o código não-instalado. Com src/, Python só encontra o pacote se ele estiver instalado (via pip install -e .), garantindo que os testes usam exatamente o mesmo pacote que o usuário final usará.
  • Separação clara: tests/, docs/, scripts/ ficam na raiz — fora do pacote. Não há risco de acidentalmente incluir arquivos de teste no pacote distribuível.

3.2 Plugin Pattern com ABCs

ABC Python docs Strategy Pattern Open/Closed Principle

O coração da arquitetura é o Plugin Pattern (também chamado de Strategy Pattern): definimos contratos abstratos que o pipeline usa, e qualquer implementação que respeite o contrato pode ser plugada sem mudar o código do pipeline.

Temos dois contratos principais:

# CONTRATO 1: Fonte de dados
class BusinessSource(ABC):
    """De onde vêm os negócios."""
    name: str = "desconhecida"

    @abstractmethod
    def fetch(self, cidade: str, admin_level: int = 8,
              limit: int | None = None) -> Iterable[RawPlace]:
        raise NotImplementedError

# CONTRATO 2: Enriquecedor de dados
class BusinessEnricher(ABC):
    """Melhora negócios já encontrados."""
    name: str = "desconhecido"

    @abstractmethod
    def enrich(self, businesses: list[Business], cidade: str) -> list[Business]:
        raise NotImplementedError

O orquestrador run_scout() trabalha apenas com as interfaces — nunca com implementações concretas:

def run_scout(cidade: str, sources: list[BusinessSource] | None = None,

Open/Closed Principle
O pipeline está aberto para extensão (posso adicionar GooglePlacesSource, CSVSource, LinkedInSource) mas fechado para modificação (o scout.py não precisa mudar). Isso é o Open/Closed Principle do SOLID, e é exatamente o que ABCs permitem.

Para adicionar uma nova fonte de dados, basta criar uma classe que herde de BusinessSource:

def run_scout(cidade, admin_level=8, limit=None,
              source=None, sources=None, enrichers=None) -> ScoutRun:

    # ── ESTÁGIO 1: COLETA ─────────────────────────────────────────────
    brutos_por_fonte = []
    for src in lista_fontes:
        raw_list = list(src.fetch(cidade, admin_level=admin_level, limit=limit))
        brutos_por_fonte.append((src.name, raw_list))

    # ── ESTÁGIO 2: DEDUPLICAÇÃO ───────────────────────────────────────
    # Primeira passagem: por (osm_type, osm_id) — exata
    vistos_ids: set[tuple[str, int]] = set()
    negocios: list[Business] = []
    for _nome_fonte, brutos in brutos_por_fonte:
        for raw in brutos:
            chave = (raw.osm_type, raw.osm_id)
            if chave in vistos_ids:
                continue
            vistos_ids.add(chave)
            negocios.append(build_business(raw))    # classifica aqui

    # Segunda passagem: por nome similar (multi-fonte)
    if len(lista_fontes) > 1:
        unicos: list[Business] = []
        for b in negocios:
            if not any(_similar(b.nome, u.nome) for u in unicos):
                unicos.append(b)
        negocios = unicos

    # ── ESTÁGIO 3: ENRIQUECIMENTO ─────────────────────────────────────
    if enrichers:
        for enricher in enrichers:
            negocios = enricher.enrich(negocios, cidade)

    # ── ESTÁGIO 4: ORDENAÇÃO ──────────────────────────────────────────
    negocios.sort(key=lambda b: b.score, reverse=True)

    return ScoutRun(cidade=cidade, fonte=fonte_str, negocios=negocios)

Similaridade de nomes: Jaccard por bigramas

Para deduplicar o mesmo negócio que aparece no OSM com nome "McDonald's" e no Serper com "McDonald's Guarujá Shopping", usamos a similaridade de Jaccard sobre bigramas de caracteres:

def _bigrams(s: str) -> frozenset[str]:
    """Gera todos os pares de caracteres consecutivos."""
    # "bar" → {"ba", "ar"}
    # "barra" → {"ba", "ar", "rr", "ra"}
    s = re.sub(r"\s+", " ", s)
    return frozenset(s[i : i + 2] for i in range(len(s) - 1))

def _similar(a: str | None, b: str | None, threshold=0.45) -> bool:
    if not a or not b:
        return False
    na, nb = _normalizar(a), _normalizar(b)
    if na == nb:
        return True                          # match exato: pulamos o cálculo
    ba, bb = _bigrams(na), _bigrams(nb)
    if not ba or not bb:
        return False
    inter = len(ba & bb)                   # intersecção dos conjuntos
    union = len(ba | bb)                   # união dos conjuntos
    # Jaccard = |A ∩ B| / |A ∪ B|
    return (inter / union) >= threshold if union else False

Threshold de 0.45: permissivo o suficiente para pegar "Drogasil" ↔ "Farmácia Drogasil" mas estrito o suficiente para não confundir "Restaurante A" com "Restaurante B".

Cada módulo tem uma responsabilidade declarada em sua docstring. Isso é o Single Responsibility Principle:

O Scout segue esse espírito: run_scout() não sabe se está sendo chamado pela CLI, por um script, ou por testes — e não precisa saber.

4.2 Type Hints e from __future__ import annotations

PEP 484 — Type Hints PEP 563 — Postponed Evaluation mypy — verificador de tipos

Todo arquivo começa com:

Este import (PEP 563) faz com que todas as anotações de tipo sejam avaliadas de forma adiada — tratadas como strings em vez de código. Dois benefícios:

  • Referências circulares: class A pode ter def metodo(self) -> B mesmo que B seja definido depois de A no mesmo arquivo
  • Sintaxe moderna no Python 3.9+: list[str] em vez de List[str], str | None em vez de Optional[str] — funciona até em Python 3.8 com o import
# Com from __future__ import annotations, isso funciona em Python < 3.10:

    ...

# Sem o import, precisaria ser:

def _similar(a: Optional[str], b: Optional[str], threshold: float = 0.45) -> bool:
    ...

def _first(tags: dict[str, str], chaves: tuple[str, ...]) -> str | None:
    """Retorna o primeiro valor encontrado entre as chaves, ou None."""
    for k in chaves:
        v = tags.get(k)
        if v:
            return v.strip()
    return None

# Uso:

Pirâmide de Testes

Martin Fowler — Test Pyramid

Seguimos a pirâmide: mais testes unitários (rápidos, isolados), menos testes de integração (mais lentos, mais dependências), e testes de performance (sem limite estrito de contagem mas com ordens de grandeza definidas):

TipoCaracterísticaTestesTempo
UnitárioUma função, sem dependências externas~50< 0.1s
IntegraçãoMúltiplos módulos juntos, com mocks de rede~15< 0.5s
PerformancePipeline completo com dados sintéticos12< 0.7s

Testes de Performance com tracemalloc

Os testes de performance estabelecem limites aceitáveis — se uma mudança futura ultrapassar um limite, o teste falha e a regressão é detectada:

from contextlib import contextmanager
import tracemalloc, time

@contextmanager
def _medir():
    """Mede tempo de parede e pico de memória."""
    tracemalloc.start()
    t0 = time.perf_counter()
    yield
    elapsed = time.perf_counter() - t0
    _, pico = tracemalloc.get_traced_memory()
    tracemalloc.stop()
    _medir.ultimo_tempo = elapsed
    _medir.ultimo_mem_kb = pico / 1024

def test_pipeline_500_negocios_abaixo_de_500ms():
    places = _raw_sem_site(500)          # dados sintéticos, sem rede
    source = _SyntheticSource(places)

    with _medir():
        run = run_scout("Teste", source=source)

    assert run.total == 500
    assert _medir.ultimo_tempo < 0.5, (
        f"Pipeline de 500 negócios demorou {_medir.ultimo_tempo:.3f}s "

Por que o DomainGuesser REDUZIU leads quentes?

Config B tem menos leads quentes (29) que o baseline (34). Isso parece contra-intuitivo — o DomainGuesser descobriu 60 sites novos, o que deveria ser bom. Mas:

Quando o DomainGuesser descobre que "Restaurante X" tem site em restaurantex.com.br, ele muda o site_status de SEM_SITE para COM_SITE. O score base muda:

  • SEM_SITE: score base = +60 (melhor oportunidade de venda)
  • COM_SITE: score base = +25 (já tem site — menos urgência)

Resultado: 60 negócios que eram "leads quentes sem site" viraram "negócios com site a auditar". O dado ficou mais correto — esses negócios realmente têm site. Para leads de venda direta, eles saem do radar imediato. Para o Agente Auditor (fase 3), eles entram — e podem ser leads de redesign.

Insight sobre o produto
O DomainGuesser não é para aumentar leads quentes — é para corrigir dados. O Auditor usará esses sites descobertos para avaliar qualidade e gerar leads de "seu site é ruim, posso melhorar". São produtos diferentes para o mesmo cliente.

Por que o Serper foi o game-changer?

O Serper adicionou 133 negócios novos que o OSM não conhecia — e quase todos vieram com telefone. Isso explica o salto de 48 para 246 contactáveis.

O OpenStreetMap é mantido por voluntários. Negócios informais, novos ou em cidades menores simplesmente não estão mapeados. O Google Maps tem equipe profissional e indexa praticamente todo negócio com CNPJ ativo.

O que pode melhorar

⚠️
httpx.Client por chamada no DomainGuesser — cada _checar_dominio() cria um novo cliente. O ideal seria um Client por thread com connection pool. Impacto atual: pequeno (a latência de rede domina o tempo). Impacto em escala: significativo.
⚠️
Dedup por nome também aplica dentro da mesma fonte — a segunda passagem de dedup por bigrama Jaccard compara todos os negócios de todas as fontes. Isso é correto para cross-source mas pode deduplicar duas filiais legítimas com nomes parecidos ("McDonald's Enseada" e "McDonald's Pitangueiras"). Ideal: só checar cross-source.
⚠️
Sem cache de domínios — o mesmo negócio tem seus domínios re-verificados toda execução. Um cache em SQLite com TTL de 7 dias economizaria 90% das requests HTTP em re-execuções.
⚠️
O A/B test roda o OSM duas vezes — configs A e B fazem queries independentes à Overpass. Os resultados são comparáveis mas não idênticos (o OSM pode retornar dados ligeiramente diferentes). Para um A/B justo, deveríamos fixar os dados de entrada.
⚠️
reporter.py não foi mostrado aqui — o gerador de HTML usa Jinja2 com um template .html.j2, mas os detalhes do template ficaram fora deste relatório. Vale um estudo separado do template para quem quiser entender a geração do dashboard.
