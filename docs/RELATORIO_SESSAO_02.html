2. O que foi implementado e por quê

2.1 DomainGuesser — descobre sites ocultos

Problema: O OpenStreetMap tem ótima cobertura geográfica mas baixa cobertura de URLs. Muitos negócios têm site, mas o dono nunca cadastrou o endereço no OSM. Você citou um exemplo real: o Hotel Ilhas da Grécia tem site em hotelilhasdagrecia.com.br, mas aparecia no Scout como "sem site".

Solução: Para cada negócio classificado como SEM_SITE, tentamos adivinhar o domínio a partir do nome e fazemos uma requisição HTTP rápida para ver se o endereço responde.

# Exemplo: "Hotel Ilhas da Grécia" gera estes candidatos:
hotelilhasdagrecia.com.br   # slug completo
hotel-ilhas-da-grecia.com.br  # variante com hifens
hotelilhasgrecia.com.br     # sem artigos (da)
hotelilhasdagrecia.com      # sem .br

Por que HTTP HEAD e não GET? O método HEAD pede ao servidor apenas o cabeçalho de resposta — o servidor confirma que o site existe sem enviar o corpo da página. Isso é ~10x mais rápido e consome zero banda.

Por que threads e não sequencial? Verificar 287 negócios um por um levaria vários minutos (latência de rede somada). Com 15 threads simultâneas o tempo cai para ~30 segundos.

2.2 Filtro de nomes genéricos (refinamento necessário)

Na primeira versão sem filtro, o DomainGuesser encontrava falsos positivos preocupantes: "Pousada" → pousada.com (um portal de pousadas, não a pousada específica em Guarujá). O domínio existia e respondia HTTP 200, mas era o site errado.

Regra implementada:

  • Se o nome for apenas uma palavra de categoria genérica ("Pousada", "Bar", "Lan House", "Bicicletaria"…) → pula sem verificar.
  • Se o nome tiver 2+ palavras significativas → verifica. ("Pousada Caiçara" → ok)
  • Se tiver 1 palavra com 8+ caracteres → provavelmente é marca, verifica. ("McDonald's" → ok, "Pousada" → não passa)

Resultado: falsos positivos caíram de 19 para 12 descobertas em 80 negócios — qualidade muito melhor.

2.3 SerperSource — Google Maps como segunda fonte

Problema: O OSM tem contato (telefone) em apenas 12,5% dos negócios. O Google Maps tem cobertura muito maior. Serper.dev é um serviço que permite consultar o Google Maps via API: 2.500 buscas gratuitas/mês sem cartão.

Estratégia: Em vez de buscar um negócio por vez (esgotaria a cota), fazemos buscas por setor: "restaurantes Guarujá SP", "salões de beleza Guarujá SP" etc. Cada chamada retorna até 20 resultados e custa apenas 1 crédito da cota. O scout completo de Guarujá gasta ~17 chamadas.

Graceful degradation: Se SERPER_API_KEY não estiver configurada, a fonte simplesmente retorna lista vazia com um aviso — o pipeline não quebra, apenas roda sem o Serper. Esta é uma boa prática de engenharia: o sistema funciona no modo mais simples, e vai melhorando à medida que você adiciona recursos.

Como ativar: Configure no arquivo .env:

SERPER_API_KEY=sua_chave_aqui

Depois rode:

fabrica scout run --com-serper

3. Cada biblioteca — o que faz e por que foi escolhida

💡

O critério de seleção foi sempre: menor dependência possível, máxima compatibilidade com Python 3.14, zero custo. Testamos a stack completa antes de começar a codificar.

httpx — requisições HTTP

Substituto moderno da biblioteca requests. Por que httpx e não requests?

  • Suporte nativo a async: em fases futuras do projeto (Auditor, Creator), vamos fazer centenas de requisições simultâneas. httpx tem AsyncClient pronto; requests não.
  • Timeout por fase: httpx permite configurar timeout separadamente para conexão, leitura e escrita — mais controle no DomainGuesser.
  • API quase idêntica ao requests: curva de aprendizado zero.

Onde usamos: Overpass API (buscas OSM), DomainGuesser (verificação de domínios), SerperSource (Google Maps).

# Exemplo real do DomainGuesser
resp = httpx.head(
    f"https://{dominio}",
    follow_redirects=True,   # segue redirecionamentos (301/302)
    timeout=3.0,             # abandona se demorar mais de 3s
)

Pydantic v2 — validação de dados com tipagem forte

Pydantic cria "moldes" para os seus dados: você descreve os campos com tipos Python, e ele garante que os dados estão no formato certo, convertendo e validando automaticamente.

class Business(BaseModel):
    nome: str | None = None         # aceita string ou None
    score: int = 0                   # sempre um inteiro
    website_kind: WebsiteKind = WebsiteKind.NENHUM  # enum com valor padrão

Por que v2 e não v1? O núcleo do Pydantic v2 foi reescrito em Rust, o que o torna 5-50x mais rápido que a v1. Isso importa quando processamos centenas ou milhares de negócios.

model_copy(): uma feature importante que usamos no DomainGuesser. Em vez de criar um objeto do zero quando atualizamos um negócio, copiamos o existente mudando só os campos necessários:

# Cria novo Business com site atualizado, mantendo todo o resto igual
negocio_novo = negocio_antigo.model_copy(update={
    "website": "https://hotelilhasdagrecia.com.br",
    "website_kind": WebsiteKind.PROPRIO,
    **nova_avaliacao,
})

Typer + Rich — CLI bonita e ergonômica

Typer transforma funções Python comuns em comandos de terminal, usando as anotações de tipo para inferir os argumentos automaticamente. Você escreve Python, ele gera o CLI.

# Typer lê os tipos e cria --cidade, --limit, --enriquecer automaticamente
def scout_run(
    cidade: str = typer.Option("Guarujá"),
    limit: int | None = typer.Option(None),
    enriquecer: bool = typer.Option(False),
)

Rich cuida da formatação visual: tabelas, painéis com borda, cores, spinner de progresso. Sem Rich, o output seria texto plano; com Rich, parece um produto profissional.

Problema que encontramos: o Windows usa encoding CP1252 por padrão, que não consegue renderizar emojis. O spinner do Rich exibiu erro na primeira rodada. A correção foi definir PYTHONIOENCODING=utf-8 antes de rodar. Isso é um problema do terminal do Windows, não do nosso código — em sistemas Linux/Mac não ocorre.

Jinja2 — geração do relatório HTML

Jinja2 é um motor de templates: você escreve um arquivo HTML com marcadores especiais {% ... %} e {{ ... }}, e ele preenche com seus dados. A alternativa seria concatenar strings Python — verboso e propenso a erros.

<!-- Trecho do template report.html.j2 -->
{% for b in top_leads %}
  <tr>
    <td>{{ b.nome or "(sem nome)" }}</td>
    <td class="score">{{ b.score }}</td>
  </tr>
{% endfor %}

Por que não f-strings? O HTML tem muitas aspas, chaves e caracteres especiais. Misturar HTML com strings Python é um pesadelo de manutenção. Jinja2 mantém o template limpo e separado da lógica.

Segurança (XSS): implementamos uma função _j() que escapa caracteres perigosos (< > &) antes de embutir dados como JSON nos <script> do relatório. Sem isso, um nome de negócio com <script> poderia injetar JavaScript no relatório.

pytest — testes automatizados

pytest encontra e roda funções que começam com test_. Simples assim. Mas o que o torna poderoso são dois recursos que usamos extensivamente:

1. parametrize — roda o mesmo teste com diferentes entradas:

@pytest.mark.parametrize("nome, esperado", [
    ("Farmácia Central", "farmaciacentral.com.br"),
    ("Hotel Ilhas da Grécia", "hotelilhasdagrecia.com.br"),
])
def test_gerar_candidatos(nome, esperado):
    assert gerar_candidatos(nome)[0] == esperado

Isso cria 2 testes com 1 função. No nosso projeto, geramos 11 testes de setor e 11 de filtro de nomes com este recurso.

2. monkeypatch — substitui funções reais por versões controladas nos testes:

def test_site_descoberto(monkeypatch):
    # Substitui a função real de rede por uma que sempre "encontra" o site
    monkeypatch.setattr(
        "fabrica_sites...domain_guesser._encontrar_site",
        lambda nome, timeout: f"https://{nome}.com.br"
    )
    # O teste roda sem nenhuma requisição de rede real

Isso é fundamental: os testes rodam em 0,37 segundos porque não fazem requisições HTTP. Sem monkeypatch, cada teste demoraria segundos e dependeria da internet.

concurrent.futures (stdlib) — paralelismo com threads

Esta é da biblioteca padrão do Python — não precisamos instalar nada extra. ThreadPoolExecutor cria um grupo de threads ("pool") e distribui o trabalho entre elas.

with ThreadPoolExecutor(max_workers=15) as pool:
    # Submete todas as verificações de uma vez
    futures = {
        pool.submit(_encontrar_site, b.nome, timeout): i
        for i, b in alvos.items()
    }
    # Processa os resultados na medida em que chegam
    for future in as_completed(futures):
        resultado = future.result()   # URL ou None

Por que threads e não asyncio? O resto do pipeline é síncrono. Misturar código async com síncrono exigiria refatorar todo o pipeline. Threads são o "caminho de menor resistência" aqui: funcionam muito bem para operações de rede (I/O-bound), onde o thread fica parado esperando a resposta — momento em que outro thread pode trabalhar.

Thread safety: cada thread cria seu próprio httpx.Client. Não compartilhamos estado mutável entre threads, então não precisamos de locks.

4. Arquitetura do código — como está organizado

O projeto segue um princípio central: cada arquivo faz uma coisa só. Isso é chamado de Single Responsibility Principle. Veja o pipeline completo:

Padrão Plugin (sources e enrichers)

Tanto as fontes quanto os enriquecedores seguem um padrão de plugin: cada um implementa uma interface simples (uma classe abstrata), e o pipeline não sabe nem se importa com os detalhes de cada implementação.

BusinessSource (interface)

class BusinessSource(ABC):
    name: str = "desconhecida"

    @abstractmethod
    def fetch(self, cidade, ...)
        -> list[RawPlace]:
        ...

Qualquer fonte de dados implementa isso

BusinessEnricher (interface)

class BusinessEnricher(ABC):
    name: str = "desconhecido"

    @abstractmethod
    def enrich(self, businesses, cidade)
        -> list[Business]:
        ...

Qualquer enriquecedor implementa isso

Benefício: para adicionar o Google Places como terceira fonte, basta criar sources/google_places.py que herda de BusinessSource e implementa fetch(). Não é necessário tocar em mais nenhum arquivo.

Layout src/ — por que não colocamos tudo na raiz?

O projeto usa o padrão src/fabrica_sites/ em vez de colocar o código na raiz. Isso parece burocrático mas tem uma razão importante:

Se o código ficasse na raiz, Python poderia importá-lo sem instalação — o que cria bugs difíceis de reproduzir. Com o layout src/, só funciona depois de pip install -e ., garantindo que todos usam o mesmo pacote instalado.

Os imports relativos (com pontos) refletem essa hierarquia:

# Dentro de enrichers/domain_guesser.py
from ....models import Business   # 4 níveis acima = fabrica_sites/
from ..scorer import score          # 2 níveis acima = agents/scout/
from .base import BusinessEnricher  # mesmo nível = enrichers/

# Mapa do pacote:
# src/fabrica_sites/agents/scout/enrichers/domain_guesser.py
#              ....          ..         .   (contando os pontos)

Como o DomainGuesser funciona por dentro (passo a passo)

Esta é a feature mais inovadora desta sessão. Veja o fluxo completo:

# Passo 1 — filtrar apenas quem vale verificar
alvos = {i: b for i, b in enumerate(businesses)
          if b.site_status is SiteStatus.SEM_SITE
          and b.nome
          and _nome_vale_tentar(b.nome)}   # filtra "Bar", "Pousada"...

# Passo 2 — paralelizar: cada thread fica responsável por um negócio
with ThreadPoolExecutor(max_workers=15) as pool:
    futures = {pool.submit(_encontrar_site, b.nome, timeout): i
               for i, b in alvos.items()}

# Passo 3 — para cada negócio, tenta candidatos em sequência (para no 1º)
def _encontrar_site(nome, timeout):
    for dominio in gerar_candidatos(nome):
        if _checar_dominio(dominio, timeout):
            return f"https://{dominio}"
    return None

# Passo 4 — para os que encontraram: atualizar Business e recalcular score
for idx, url in encontrados.items():
    result[idx] = _rescore(result[idx], url)
    # _rescore usa model_copy + scorer.score() para gerar nova instância

Por que o score CAI quando encontramos um site? O score base de SEM_SITE é 60 pontos. Se descobrimos que o negócio tem site, ele vira COM_SITE com base de 25 pontos — porque ele não é mais um prospect primário para venda de site, e o Auditor terá que avaliar a qualidade do site existente. Os dados ficam mais precisos.

Deduplicação de duas fontes (OSM + Serper)

Quando usamos Overpass e Serper juntos, o mesmo negócio pode aparecer nas duas fontes com IDs diferentes. Resolvemos isso com dois níveis de deduplicação:

  1. Por ID OSM: (osm_type, osm_id) — rápido, O(n), elimina duplicatas dentro de uma mesma fonte.
  2. Por similaridade de nome: compara cada novo negócio com todos os já aceitos. Se a similaridade de bigramas for ≥ 45%, descarta o duplicado.

O que são bigramas? Pares de caracteres consecutivos. "Hotel" vira: "Ho", "ot", "te", "el". Comparar bigramas é mais robusto que comparar strings diretamente — funciona mesmo com palavras na ordem trocada ("Drogasil Guarujá" ≈ "Guarujá Drogasil").

# Jaccard de bigramas: |A ∩ B| / |A ∪ B|
def _similar(a, b, threshold=0.45):
    ba = _bigrams(_normalizar(a))   # frozenset de bigramas normalizados
    bb = _bigrams(_normalizar(b))
    inter = len(ba & bb)
    union = len(ba | bb)
    return (inter / union) >= threshold

Limitação honesta: esta deduplicação é O(n²) — cada novo negócio é comparado com todos os anteriores. Para 385 negócios isso é rápido (<1ms). Para 10.000 negócios precisaríamos de uma estrutura de índice (ex.: MinHash LSH). Anotado para escala futura.

8. Bug descoberto e corrigido — análise detalhada

🐛 Bug: falso positivo no detector de redes sociais

Onde: classifier.py, função _is_social(url)

Impacto: qualquer negócio cujo site contivesse "x.com" como substring era incorretamente classificado como "só rede social" — mesmo sendo um site próprio legítimo. Ex.: "salaox.com.br", "maxcompra.com.br", "faxe.com.br".

O código errado

- def _is_social(url: str) -> bool: - u = url.lower() - return any(dom in u for dom in _SOCIAL_DOMAINS) - # "x.com" in "https://salaox.com.br" → True ← BUG!

O código corrigido

+ def _is_social(url: str) -> bool: + try: + netloc = urlparse(url).netloc.lower() + host = netloc.removeprefix("www.") + # Compara o HOST, não a URL inteira + return any(host == dom or host.endswith("." + dom) + for dom in _SOCIAL_DOMAINS) + except Exception: + return False # fallback seguro

O que mudou: usamos urlparse() para extrair apenas o host da URL (ex.: "salaox.com.br") e comparamos se o host é igual a "x.com" ou termina com ".x.com". A string "salaox.com.br" não é igual a "x.com" nem termina com ".x.com" — classificação correta.

Como foi encontrado: um teste escrito especificamente com a URL "https://salaox.com.br" falhou ao verificar que o negócio teria site_status = COM_SITE. Sem o teste, este bug poderia ter passado para produção sem ser detectado.

9. Autoavaliação — análise honesta do que foi feito

🪞

Esta seção avalia o trabalho com os olhos de quem vai manter e evoluir o código — sem inflar nem diminuir.

O que ficou bem

  • DomainGuesser é inovador e funciona: a ideia de tentar "nomenegocio.com.br" é simples, não custa nada e descobriu 60 sites reais em Guarujá. É uma abordagem que não vi em projetos similares de scraping de leads.
  • Arquitetura plugável: adicionar uma terceira fonte de dados (ex.: Google Places) não requer tocar em scout.py, cli.py nem nos testes existentes. Basta criar um novo arquivo em sources/.
  • Testes de qualidade: os testes não só verificam que o código funciona, mas também definem o comportamento esperado de forma clara. O caso do "lead quente exige contato" e o bug do x.com são exemplos de testes que encontraram problemas reais.
  • Graceful degradation: o sistema funciona em modo básico sem nenhuma API key, vai melhorando progressivamente conforme você adiciona chaves. Esta filosofia garante que o custo inicial é zero.
  • Bug do classifier encontrado e corrigido: sem o teste, este bug estaria em produção silenciosamente reclassificando sites legítimos como redes sociais.

O que tem espaço para melhorar

  • DomainGuesser ainda tem falsos positivos de redes nacionais: McDonald's, Burger King e Receita Federal foram "descobertos" com seus sites nacionais — corretos tecnicamente, mas não são prospects para venda de site local. Uma melhoria futura seria detectar domínios de redes nacionais (lista de franquias conhecidas) e filtrá-los.
  • Negócios repetidos (filiais): "Oggi Sorvetes" aparece 7 vezes e "Drogaria São Paulo" aparece 3 vezes (filiais diferentes no OSM). O pipeline trata cada filial como negócio separado, o que é correto para o Scout, mas para o Prospector será necessário consolidar por rede/CNPJ para não abordar a mesma empresa várias vezes.
  • httpx.Client criado por thread: no DomainGuesser, cada chamada a _checar_dominio() cria um novo httpx.Client (e portanto um novo pool de conexões TCP). Seria mais eficiente criar um cliente por thread e reutilizá-lo. O impacto é pequeno para 300 negócios, mas relevante em escala.
  • Sem cache entre execuções: o DomainGuesser refaz todas as verificações HTTP a cada execução do Scout. Se você rodar o Scout amanhã, verificará os mesmos 300 domínios de novo. Uma tabela domain_cache no SQLite resolveria: verifica só domínios que nunca foram checados ou que foram checados há mais de N dias.
  • O comparativo do script executa o OSM duas vezes: o script comparativo_enricher.py chama run_scout() duas vezes independentemente. O segundo fetch pode retornar resultados ligeiramente diferentes do primeiro (OSM pode ser atualizado, ordem de elementos não é garantida). Um comparativo perfeito passaria o mesmo run_base pelo enricher, sem buscar os dados duas vezes.

10. Próximos passos

Com base no que aprendemos, aqui está a ordem recomendada:
